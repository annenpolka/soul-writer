# Part 1: プロジェクト文脈・意思決定・現状

---

## 背景と動機

### なぜLLMOps評価層か

- ユーザーはLLMOps領域の実務スキルを身につけたい（転職活動中、AI Ops系ポジションに関心）
- soul-writerは既に大量の生成済み作品（2,162本）を持つLLMベースの小説生成パイプラインである
- 既存の評価システム（Judge 8軸 + Reader Jury 5ペルソナ）は存在するが、verdictの94%がpublishableで6%がexceptionalとなり、実質的にフィルタリングとして機能していない
- ユーザー自身も「評価部分はあまり当てにしていない」と明言

### 目的の明確化

**やりたいこと**: 既存コーパスから「良い作品」を特定し、良い作品の再現性を高めるためのフィードバックループを構築する

**やらないこと（少なくとも今は）**:
- 既存のJudge/Reader評価システムの改善
- プロンプトバージョンと品質の因果分析（交絡があり信頼度が低い）
- ルーブリックベースの精緻なスコアリング

---

## 重要な設計判断

### 1. 人間ラベリングが起点

LLM-as-a-Judgeのキャリブレーションには「人間の正解ラベル」が必要。ユーザー自身がドメインエキスパートであり、「良い/普通/ダメ」の直感的判断を信頼する。

### 2. ゴールデンデータセットの構築が最優先

プロンプトが随時調整されてきたため、メタデータ（narrative_type等）と品質の相関分析は交絡で汚染される。しかし**テキスト品質のラベリング自体は交絡と無関係**——どのプロンプトで生成されたかに関係なく、テキストそのものの品質にラベルを付けているから。

ゴールデンデータセットがあると何ができるか:
- **evalの基準点**: 新しいプロンプトで生成した作品を、ゴールデンセットと比較して「この水準に達しているか」を判定できる
- **良い作品のテキスト特徴を抽出する素材**: 因果（なぜ良いか）ではなく記述（良い作品に共通する構造的特徴は何か）を取り出す
- **Anti-Soulの逆版**: 「こう書くな」の負例集に対する「こう書け」の正例集

### 3. 段階的アプローチ

1. 既存コーパスから◎を選んでゴールデンデータセットを作る（**現在ここ**）
2. ゴールデンセットのテキスト特徴を分析して、現行プロンプトに反映する
3. 新プロンプトで少量生成→ゴールデンセットを基準にevalを回す
4. 3のサイクルを繰り返す（Eval-driven Development）

### 4. 交絡への対処方針

既存コーパスでの因果分析よりも、**新しいプロンプトで意図的にA/Bテストをやる**方が変数が制御された状態でのフィードバックが得られる。これはevalのCI/CDパイプライン——プロンプト変更を「品質ゲート」に通すワークフローそのもの。

## 現在の状態

### 作成済みのツール

**soul-labeler.html** — スタンドアロンのラベリングUI
- ブラウザで開いてJSONファイルを読み込んで使う
- テイスティングビュー（冒頭500字・中盤500字・終盤300字）と全文表示の切替（`f`キー）
- キーボード操作: `1`=◎, `2`=△, `3`=✗, `s`=skip, `←`=戻る
- ◎と✗は一言メモ入力あり（Enter確定、Escスキップ）
- バイアス排除のため、既存verdict/emotion/narrative_type等のメタデータは表示しない
- 順序はハッシュベースでシャッフル済み（生成時期の系列相関を排除）
- localStorageで進捗自動保存、中断・再開可能
- エクスポートでラベル結果＋隠しメタデータのJSONダウンロード

**データファイル**:
- `tasting-data-full.json`（85MB）— 全文入り、2,162作品
- `tasting-data.json`（9MB）— テイスティングビューのみ

## 次のステップ

### Phase 1: ラベリング（ユーザーが手動で実施中 or これから実施）

soul-labeler.htmlを使って作品にラベルを付ける。目標は最低20-30本の◎。

### Phase 2: ゴールデンデータセット分析

ラベリング結果のJSONを受け取ったら:

1. **メタデータ相関分析**: ◎/△/✗とnarrative_type, emotion, toneの分布を可視化（交絡の注意付き）
2. **テキスト特徴抽出**: ◎群と✗群のテキスト構造的差異を分析
   - 冒頭の構造（身体的行動が何行目で出るか）
   - 対話と地の文の比率
   - 感覚描写の密度と配置
   - aesthetics-creedの原則（感情のラベリング回避、二重記述回避等）の遵守度
3. **Anti-Soulの逆版**: ◎群から「こう書け」の正例パターンを抽出

### Phase 3: プロンプトフィードバック

分析結果をsoul-writerのプロンプトに反映:
- `src/prompts/agents/writer.yaml` — writerの指示
- `src/prompts/sections/aesthetics-creed.yaml` — 美学的信条
- `soul/fragments/` — 参照断片にゴールデンな断片を追加
- `soul/constitution.json` — 必要に応じてルール追加

### Phase 4: Eval-driven Development

- 新プロンプトで少量生成（10-20本）
- DeepEval（G-Eval）でゴールデンセットを基準にしたLLM-as-a-Judge評価
- 品質ゲートとしてCI/CDに統合（vitest + DeepEval or カスタムevalスクリプト）
- Langfuseでトレーシング（オプション）

# Part 2: LLMOps現状リサーチ（2026年4月時点）

## 市場と構造

LLMOps市場は2025年の約52億ドルから2032年には約198億ドルへ成長予測（CAGR 21.3%）。LLMOpsは「モデルをAPIの後ろに置く」段階を超え、プロンプト管理・ガードレール・評価・オブザーバビリティ・コスト制御・ガバナンスを含む完全なプロダクションスタックへ進化した。

### MLOps → LLMOpsの構造的分岐

| 側面 | 従来のMLOps | LLMOps |
|:-----|:-----------|:-------|
| データ形式 | 構造化データ | 自然言語（非構造化） |
| 出力の性質 | 決定論的・予測可能 | 非決定論的・文脈依存 |
| 評価基準 | 精度・適合率・再現率 | ハルシネーション率・忠実性・毒性・ユーザー満足度 |
| トレーニングコスト | 主要コスト要因 | 低い（ファインチューニング中心） |
| 推論コスト | 比較的低い | 主要コスト要因（GPU依存・トークン課金） |
| 主要な課題 | モデルドリフト | プロンプトインジェクション・ハルシネーション・コスト爆発 |

### LLMOpsの7段階ライフサイクル

1. ユースケース定義・モデル選択
2. プロンプトエンジニアリング・バージョニング（LangSmith, PromptLayer）
3. デプロイ・統合（Bifrost, Portkey, LiteLLM）
4. モニタリング・オブザーバビリティ（Arize Phoenix, Langfuse, W&B Weave）
5. 評価・品質保証（RAGAS, DeepEval, Giskard）
6. コスト・レイテンシ最適化（Redis LangCache, vLLM）
7. ガバナンス・コンプライアンス・インシデント対応

このライフサイクルは連続的——モデルプロバイダーが基盤モデルを更新し、現実世界の入力が変化し、ビジネス要件が進化するため、静止状態が存在しない。

## エージェント・オーケストレーション

2026年にはMCP（Model Context Protocol）がLinux Foundation傘下となりデファクト標準化。120以上のエージェントAIツールが11カテゴリで展開。しかしプロダクション・インフラはフレームワークに大きく遅れている。

GetOnStackの事例: マルチエージェントシステムのエージェント間無限ループにより、週127ドルの運用コストが4週間で47,000ドルに膨れ上がった（11日間未検出）。メッセージキュー、分散トレーシング、コスト監視、サーキットブレーカー等のプロダクション・インフラが未成熟であることを示す。

---

## 推論コスト（"LLMflation"）

a16z分析: 同等性能のLLMの推論コストは年間10倍のペースで低下。GPT-3（2021年）が100万トークンあたり60ドル→同等性能モデルが現在0.06ドル。3年で1,000倍。

要因: 量子化（16bit→4bit）、ソフトウェア最適化、効率的小規模モデル、改善されたインストラクションチューニング。

Kubernetesクラスタ上でのLLMワークロードは戦略的最適化により推論・トレーニングコストの最大60%削減が報告されている。

---

## 規制環境

EU AI Actの主要義務が**2026年8月2日**に発効。ハイリスクAIシステムにはリスク管理、データガバナンス、技術文書、ロギング、人的監視、適合性評価、CEマーキング、EUデータベース登録が義務付け。違反は最大3,500万ユーロまたは全世界年間売上高の7%。

コンプライアンスは「あとから貼り付けるもの」ではなく「設計時点から組み込むエンジニアリング」に。AIゲートウェイをコントロールプレーンとして使用し、すべてのモデルトラフィックを一元的に制御・記録する方向。

# Part 3: 評価層深掘りリサーチ

---

## 評価メトリクスの3世代

### 第1世代: 統計的スコアラー（BLEU/ROUGE）

n-gramオーバーラップ測定。表層的な文字列比較であり、意味的正しさを捉えられない。LLMアプリケーション評価には構造的に不適合だが、構造化された要約タスクのベースライン比較には使われる。

### 第2世代: モデルベース・スコアラー（SummaC, QAFactEval等）

NLIモデルで参照なしにテキスト品質を評価。第1世代より改善だが、ドメイン固有の判断には限界。

### 第3世代: LLM-as-a-Judge（現在の主流）

強力なLLMに評価プロンプトとルーブリックを与え、別のLLMの出力を評価させる。人間レビューの500-5,000倍のコスト削減、人間の判断との一致率は約80%（人間同士の一致率と同等）。

**既知のバイアス**:

| バイアス | 影響度 | 対策 |
|:--------|:------|:-----|
| 位置バイアス | GPT-4で40%不一致率 | (A,B)と(B,A)両順序で評価、一貫した結果のみカウント |
| 冗長性バイアス | 約15%スコア膨張 | 1-4スケール、簡潔さを明示的に報酬 |
| 自己強化バイアス | 同一モデルファミリーで5-7%ブースト | 異なるモデルファミリーをジャッジに使用 |
| ドメインギャップ | 専門領域で一致率10-15%低下 | スクリーニングのみ、最終判断は人間 |
| ジャッジ・ドリフト | APIアップデートで挙動変化 | バージョン固定、キャリブレーションチェック |

高リスク判断: 3-5モデル多数決でバイアス30-40%削減（コスト3-5倍）。

**コスト**: GPT-4o-miniジャッジで5メトリクス×1テストケースあたり$0.001-0.003。200問ゴールデンデータセットの1回評価は$1未満。

## RAG評価: RAGASフレームワーク

RAGパイプライン評価の事実上の標準。4つのコアメトリクス:

- **Faithfulness（忠実性）**: 回答が検索ドキュメントと事実的に一致しているか。RAG固有の最重要メトリクス
- **Answer Relevancy（回答関連性）**: 回答が元クエリにどれだけ関連しているか
- **Context Precision（コンテキスト精度）**: 関連チャンクがランキング上位に配置されているか
- **Context Recall（コンテキスト再現率）**: 関連情報が検索ドキュメントにカバーされているか

実運用: 50問の丁寧にキュレーションされた質問で安定したメトリクス。回帰テストには200-500問。

*soul-writerプロジェクトではRAGは使用していないが、「参照ベースの品質評価」のアーキテクチャとして参考になる。*

## エージェント評価

業界コンセンサスが存在しない最も未成熟な領域。

### 従来のLLM評価との根本的差異

- **非決定性の伝播**: 各ステップで不確実性が蓄積、同じ入力から全く異なる実行パスが生成
- **行動ベース評価の必要性**: タスク「完了」でも行動的に不適切な場合がある（例: ポリシーチェックをスキップしてタスク成功）
- **中間地点での失敗**: 10ステップワークフローの任意の地点で失敗しうる

### エージェント評価の4つの柱（Amazon事例）

1. **タスク成功率**: 最終的に正しい結果を出したか
2. **推論の一貫性**: 意思決定チェーンは論理的に一貫しているか
3. **ツール使用の適切性**: 正しいツール選択、正しいパラメータ、適切なレスポンス処理
4. **コスト・パフォーマンスのバランス**: トークン・API呼び出し・時間の消費量

### 最前線ツール

- **Virtue AI Agent ForgingGround**: 50以上のプロダクション環境をシミュレーション、1,000以上のアルゴリズムで自動レッドチーミング
- **LangSmith Insights Agent**: プロダクショントレース自動分析、Multi-turn Evals
- **AgentX AgentBeats**（UC Berkeley）: ベンチマーク自体をエージェント化する競技（賞金$1M超）

## 評価のCI/CD統合

### コードベース・アサーション vs LLM-as-a-Judge

- **コードベース**: 決定論的タスク向け。正解が1つ。低コスト・高頻度実行。コミットごとの回帰テストに適する。**原則: コードで検証できる失敗には常にコードベースのevalを使う**
- **LLM-as-a-Judge**: 主観的判断向け。正解が1つではない。ドメインエキスパートのラベルをLLMジャッジに自動化

### DeepEvalのpytestライクなパラダイム

LLM版pytest。G-Evalメトリクスで任意のカスタム基準のLLM-as-a-Judgeを構築可能。

提供メトリクス: G-Eval、DAG、Faithfulness、Contextual Recall/Precision/Relevancy、Hallucination、Bias、Knowledge Retention、Conversation Completeness等。

Confident AIがDeepEvalのホスト版を提供し、回帰スイート・A/Bテスト・CI/CD品質ゲート統合を実現。

### 評価パイプラインの典型フロー

```
[開発時]
プロンプト/モデル/パイプライン変更
  → ゴールデンデータセット（50-200問）に対するExperiment評価
  → ベースラインとの比較 → 品質ゲート通過/不通過
  → 通過 → 本番デプロイ

[本番運用時]
ライブトラフィックのサンプリング
  → Observation/Traceレベルの非同期LLM-as-a-Judge評価
  → メトリクス・ダッシュボード
  → アラート → 閾値超過で調査
  → 障害トレースからゴールデンデータセットにケース追加
  → [開発時]へフィードバック
```

このフライホイールが継続的評価の本質。プロダクションの障害が評価データセットを豊かにし、評価がプロダクションの品質を改善する。

## トレーサビリティ

2026年の評価スタックで最も重要視されているのはトレーサビリティ——特定の評価スコアを、それを生成した正確なプロンプトバージョン・モデルバージョン・データセットに結びつける能力。

W&B Weaveは `@weave.op` デコレータで自動トレースツリーキャプチャ。Langfuseはプロンプトバージョニングとトレーシングを統合。

## 展望（評価層固有）

- **自己評価LLM**: モデルが内部基準で自身の出力をスコアリングする方向。外部ジャッジのコスト・レイテンシ削減
- **ベンチマークのエージェント化**: 静的ベンチマークの限界を動的評価環境で突破（AgentX AgentBeats）
- **Eval-driven Development**: TDDのLLM版。評価基準を先に定義→それを通過するプロンプト/パイプラインを実装。CaO（Constraint-as-Output）と構造的に同型
- **評価コストの民主化**: GPT-4o-miniジャッジ + ローカルモデル（Ollama/vLLM）による完全オンプレミス評価
